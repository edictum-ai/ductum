# Strangler Roadmap

Ordering rationale: **security-first** (a rollback must never re-open a hole), then **foundation primitives** the autonomy/UI/extensibility work all depend on, then **autonomy + legibility**, then **UI/design reconcile**, then **extensibility + DX**. Each phase names the **one dogfood flow** that proves it. No phase ships an abstraction whose justifying flow does not yet exist.

---

## Phase 0 — Security floor (close the two holes first)
**Goal:** the two material defects — host-env secret leak and non-idempotent evidence write — are closed before any structural refactor, so every later rollback lands on a safe baseline.
**Touches dispositions:** REDESIGN `claude.ts:187` / `codex-mcp-config.ts:29` (`...process.env` → allowlisted mount); REDESIGN `evidence.ts:75-82` (transactional, idempotency-keyed append + `content_sha` side column + `UNIQUE(run_id, idempotency_key)`); REUSE FactorySecretResolver/crypto/redaction (consumed at dispatch now).
**Work:** ScopedSecretBroker materializes an allowlisted env from the encrypted FactorySecret store at dispatch, mounted into the worker, torn down at run end; ship in *allow-then-warn* mode for one dogfood cycle (log which previously-ambient vars an agent reads) then enforce. ANTHROPIC_API_KEY becomes a named FactorySecret injected only when a harness capability descriptor declares it. EvidenceLedger transactional `commitGate` (evidence + gate_evaluations + `setStage()` in one tx); confirm `SqliteStorageBackend` writes are synchronous/uncommitted-until-tx by reading edictum-ts source first (fall back to savepoint-per-commitGate if Edictum batches). Keep the `refreshRunFromWorkflow` 'done' guard exactly where it is.
**Dogfood flow:** run a real attempt to completion; grep the spawned env to prove no ambient host secret leaked; kill the process mid-`implement` and replay the gate evidence write — it is a no-op, not a duplicate or throw.

## Phase 1 — Foundation primitives (the Sealed Job Bundle seam)
**Goal:** one wire contract + durable liveness + composed dispatch, with no behavior change.
**Touches dispositions:** REUSE+EXTEND `AttemptRuntimeSnapshot` → `SealedJobBundle` (superset adapter adds the 8 fields with safe defaults; required only after the live flow is green); REDESIGN dispatcher 6-level inheritance → 6 composed services via `DispatchContext`; REUSE→REHOME `activeSessions` Map behind `LiveRunRegistry` / `LocalExecutionBackend`; NEW Lease+Fence ledger rows; NEW `bundleHash` echo on evidence writes.
**Work:** mechanical promotion of dispatcher methods to injected services, guarded by the existing dispatch suite as a characterization harness — **before** any new field lands so a regression is unambiguously attributable. Durable claim = leased run + fencing token + heartbeat (renew on existing heartbeat cadence, 2x-heartbeat expiry); live `HarnessSession` becomes a rebuildable cache. schemaVersion compat gate with a boot-newer-against-older-bundle fixture test. Logical control/data-plane invariant enforced in place (no physical package split yet).
**Dogfood flow:** attempt start → spawn → gate_check → complete routed through `SealedJobBundle` + `LiveRunRegistry` + composed Spawner; then restart the process and confirm "what is running now" is reconstructed from the durable claim, not lost with process memory.

## Phase 2 — Autonomy + legibility (resume, quarantine, whatToDoNext)
**Goal:** the factory survives a crash by resuming from the last committed gate under a fenced lease, quarantines poison tasks, and exposes one derived next-action.
**Touches dispositions:** NEW Checkpoint store (per-stage, keyed by idempotent gate-commit); NEW Reconciler (converges externalized vs live state); NEW AutonomySupervisor (wraps KEEP `cycleOnce()`); NEW `quarantined` terminal state; DELETE dead `tryReattach` interface + overstated startup log; REUSE RunStateMachine/DAG/WatcherManager unchanged.
**Work:** fence-guard only the four recovery-critical writes (terminal transitions, checkpoint commits, evidence commits, cost records). Content-derived ids for replayable evidence kinds with `INSERT ... ON CONFLICT DO NOTHING`; monotonic ids stay for append-only operator notes. `push`/`merge` stages classified rollback-required (never forward-replay an irreversible side effect); reconciler falls back to today's fresh-Run path when the worktree fails a cleanliness probe (resume is an optimization over a safe baseline). `whatToDoNext` total over a fixture of every run shape with an exhaustiveness test.
**Dogfood flow:** kill a Codex attempt mid-`implement`, restart, watch it resume from the durable checkpoint at $0-replay cost rather than restart from `understand`; force a deterministic failure and watch it land in `quarantined` as an inbox item, out of the ready queue.

## Phase 3 — UI / design-system reconcile (brand-true, inbox-first, parity)
**Goal:** collapse the dual design systems into one token layer, reskin to the brand-true dark terminal, make Home the inbox, wire intervention controls.
**Touches dispositions:** REDESIGN `stage-display.ts` literal Tailwind class maps → token-driven `statusTone`; REDESIGN dual `signal/` + `ui/` → one CSS-custom-property vocabulary (keep shadcn/Radix primitives behind the bridge — do not rewrite them); REUSE routing shell + 40 component tests; REMOVE TreeNavigator/RelativeTime/legacy `/specs` and `/agents`; rename `ProjectList.tsx` → `Home.tsx`.
**Work:** token-unification (statusTone migration) **before** the visual flip; grep literal `bg-`/`text-`/hex first. Flip `:root` to near-black `#111318`, swap orange `#ff9d5c` → signal-blue `#2F6FED` in both themes; swap Geist → Inter + Archivo Expanded (display only, via `--font-display`) + JetBrains Mono, pinned exact versions with ≥7-day publish buffer. Inbox spine on Home (deduped needs-you queue, next-command per item, prioritized like the command palette) + keep Activity/Approvals/Repair as deep views. Intervention controls (approve/cancel/pause/resume/retry/redirect) wired to run-control API + SSE (D146). Cost strip renders "unattributed" until the cost ledger fixes silent-$0 Codex. Parity conformance check scoped to existence + identifier consistency.
**Dogfood flow:** an operator opens Home, sees a blocked run in the inbox with its next-command, approves it from the inbox, and follows the per-run evidence timeline live — same action available identically in CLI and UI.

## Phase 4 — Extensibility + DX (extension seam, sandbox ladder, doctor/onboard)
**Goal:** a third executor is an adapter not a fork; a fresh operator gets zero-config start, a doctor that names the broken prerequisite and the fix, and an authoring contract agents can trust.
**Touches dispositions:** REDESIGN `BUILT_IN_HARNESSES` array + Telegram-only NotificationBackend → typed `@ductum/extension-sdk` ExtensionRegistry (closed, deny-by-default, capability-validated, operator-allowlisted; built-ins register through the same seam); NEW `SandboxDriver` interface + host driver now, container driver as the final proof step; NEW `ductum onboard` / `ductum doctor` commands (doctor reuses the readiness engine; repair stays the remediation verb); NEW generated `llms.txt`/`llms-full.txt` from live zod/TS validators with a CI drift check; NEW StateStore/Queue boundary (SQLite/in-process impl only, 1:1 wrapper, no PgQueue); REMOVE OpenCode family (7 files + plugin) in the SAME staged change as the DB CHECK-constraint repair migration, with an upgrade-clean migration test.
**Work:** the skill drives the shipped command; onboard writes through the API (DB-as-truth), prints nothing to paste. `ductum doctor --json` wired into the smoke-onboarding gate so the floor is CI-enforced. Container `SandboxDriver` ships last as the seam-proof (second driver registers behind the same interface).
**Dogfood flow:** add a second harness/agent via the dashboard, onboard a sibling repo with `ductum onboard`, and author a valid WorkflowProfile from `llms.txt` alone — the three exact D161 scenarios that previously failed — then register a container `SandboxDriver` behind the same seam without forking.

---

**Cross-phase sequencing rationale:** P0 before P1 because the dispatcher recomposition (P1) touches the D22/D24/D25-critical spawn path; sequencing security wins first means a rollback never re-opens the leak. P1 before P2 because checkpoint/resume/fencing consume the bundle's `jobAttemptId`/`fencingToken`/`pinnedSha` and the durable claim. P2 before P3 because `whatToDoNext` is the single backing for the inbox/run-view/notifications the UI renders. P3 before P4 because the extension seam and onboard CRUD bind to the unified operator-contract types the UI work settles. Physical package split, PgQueue, plugin loader, and cross-run hash-chain are all deferred — no current dogfood flow justifies them.
